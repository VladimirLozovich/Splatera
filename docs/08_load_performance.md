> *Примечание: Так как приложение Splatera работает локально и не использует сеть, оценка нагрузки производится только на файловую систему и процессор.*

## Сценарии нагрузки на систему

| S1 (Пиковая I/O нагрузка)                                                                                                                                                         | S2 (Нагрузка на RAM/GPU)                                                         | S3 (Частота запросов к локальной БД)                                                                      |
| --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| Массовый импорт. Пользователь перетаскивает папку с 1000+ фотографиями высокого разрешения. Приложению нужно прочитать все пути, сгенерировать 1000 превью и извлечь 1000 палитр. | Скроллинг сетки (Masonry Grid), содержащей 5000+ элементов, с активным фильтром. | Быстрый ввод текста в строку поиска по тегам (каждое нажатие клавиши инициирует поиск по локальной базе). |

## Оценка частоты операций (Худший случай)

При массовом импорте (S1): Генерация одного превью (resize) и извлечение цвета занимает в среднем ~200мс на CPU.

В 1 поток: 1000 фото = ~3 минуты непрерывной загрузки CPU.

Поиск (S3): При скорости печати 300 симв/мин генерируется до 5 запросов к БД в секунду.

## Меры защиты и оптимизации

**Очередь задач (Job Queue) для S1:**

*   Импорт обрабатывается асинхронно в фоне.
*   Ограничение пула потоков (Thread Pool): не более `N`
 одновременных задач обработки изображений (где `N` = количество ядер CPU минус 1, чтобы ОС не зависала).

**Защита UI от подвисаний (S2):**

Применяем Windowing / Virtual Scrolling. В DOM находятся только видимые карточки + буфер в 20%.


**Debounce (S3):**

Искусственная задержка (Debounce) на ввод текста в поиске — `300ms`. Локальный запрос в базу уходит только если пользователь перестал печатать.

**Блокировки:** 

Запрет повторного добавления одной и той же папки, пока предыдущая операция "Importing..." не завершена.
